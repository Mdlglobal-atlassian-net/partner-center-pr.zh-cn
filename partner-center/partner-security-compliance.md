<span data-ttu-id="e781d-181">了解当前的实施方案是否仅在特定的条件下强制实施 MFA。</span><span class="sxs-lookup"><span data-stu-id="e781d-181">Understand whether current implementation only enforce MFA under specific conditions only.</span></span> <span data-ttu-id="e781d-182">某些 MFA 解决方案具有灵活性，允许仅在满足特定的条件（例如，用户从未知的设备或位置进行访问）时</span><span class="sxs-lookup"><span data-stu-id="e781d-182">Some MFA solutions provide flexibility to only enforce MFA when certain conditions are met.</span></span> <span data-ttu-id="e781d-183">才强制实施 MFA。</span><span class="sxs-lookup"><span data-stu-id="e781d-183">For example, user is accessing from unknown device or unknown location.</span></span> <span data-ttu-id="e781d-184">如果为某个用户启用了 MFA，但该用户在访问合作伙伴中心时不一定要完成 MFA 验证，则可能会导致指标不是 100%。</span><span class="sxs-lookup"><span data-stu-id="e781d-184">A user, who is enabled for MFA but isn't required to complete MFA verification when accessing Partner Center, can cause the metrics to not be 100%.</span></span>

>[!NOTE]
><span data-ttu-id="e781d-185">对于已使用 Azure AD 安全性默认设置实施了 MFA 的合作伙伴，请务必注意，对于非管理员用户帐户，会根据风险强制实施多重身份验证。</span><span class="sxs-lookup"><span data-stu-id="e781d-185">For partners who have implemented MFA using Azure AD security defaults, it is important to note that for non-admin user accounts multi-factor authentication will be enforced based on risk.</span></span> <span data-ttu-id="e781d-186">仅当用户尝试进行有风险的登录（例如，从不同的位置登录）时，系统才会提示他们完成 MFA。</span><span class="sxs-lookup"><span data-stu-id="e781d-186">Users will be prompted for MFA only during risky sign-in attempts (e.g., user is signing in from a different location).</span></span> <span data-ttu-id="e781d-187">此外，用户有长达 14 天的时间来注册 MFA。</span><span class="sxs-lookup"><span data-stu-id="e781d-187">In addition, users will have up to 14 days to register for MFA.</span></span> <span data-ttu-id="e781d-188">在这 14 天内，未完成 MFA 注册的用户不会受到 MFA 验证的质询。</span><span class="sxs-lookup"><span data-stu-id="e781d-188">Users who have not complete MFA registration will not be challenged for MFA verification during the 14-day period.</span></span> <span data-ttu-id="e781d-189">因此，对于已使用 Azure AD 安全性默认设置实施了 MFA 的合作伙伴，指标可能不是 100%，这在预料之中。</span><span class="sxs-lookup"><span data-stu-id="e781d-189">Therefore, it is expected that the metrics may not be 100% for partners who have implemented MFA using Azure AD security defaults.</span></span>

### <a name="are-you-using-3rd-party-mfa-solution"></a><span data-ttu-id="e781d-190">是否在使用第三方 MFA 解决方案？</span><span class="sxs-lookup"><span data-stu-id="e781d-190">Are you using 3rd party MFA solution?</span></span>

<span data-ttu-id="e781d-191">如果使用第三方 MFA 解决方案，请确定如何将它与 Azure AD 集成。</span><span class="sxs-lookup"><span data-stu-id="e781d-191">If you are using 3rd party MFA solution, identify how you are integrating it with Azure AD.</span></span> <span data-ttu-id="e781d-192">一般情况下，有两种方法：联合与自定义控制：</span><span class="sxs-lookup"><span data-stu-id="e781d-192">In general, there are two methods, including federation and custom controls:</span></span>

* <span data-ttu-id="e781d-193">**标识联合** - 当 Azure AD 收到身份验证请求时，Azure AD 会将用户重定向到联合的标识提供者进行身份验证。</span><span class="sxs-lookup"><span data-stu-id="e781d-193">**Identity federation** - When Azure AD receives an authentication request, Azure AD will redirect the user to the federated identity provider for authentication.</span></span> <span data-ttu-id="e781d-194">身份验证成功后，联合的标识提供者会将用户重定向回到 Azure AD，并提供一个 SAML 令牌。</span><span class="sxs-lookup"><span data-stu-id="e781d-194">Upon successful authentication, the federated identity provider will redirect the user back to Azure AD along with a SAML token.</span></span> <span data-ttu-id="e781d-195">为使 Azure AD 能够识别到该用户在对联合的标识提供者进行身份验证时已完成 MFA 验证，该 SAML 令牌必须包含 *authenticationmethodsreferences*  声明（值为 *multipleauthn*）。</span><span class="sxs-lookup"><span data-stu-id="e781d-195">In order for Azure AD to recognize that the user has completed MFA verification when authenticating to the federated identity provider, the SAML token must include the *authenticationmethodsreferences* claim (with value *multipleauthn*).</span></span> <span data-ttu-id="e781d-196">检查联合的标识提供者是否支持发出此类声明。</span><span class="sxs-lookup"><span data-stu-id="e781d-196">Check whether that the federated identity provider supports issuing such a claim.</span></span> <span data-ttu-id="e781d-197">如果是，请检查联合的标识提供者是否已配置为提供此项支持。</span><span class="sxs-lookup"><span data-stu-id="e781d-197">If so, check whether the federated identity provider has been configured to do so.</span></span> <span data-ttu-id="e781d-198">如果缺少声明，Azure AD（因而也包括合作伙伴中心）将不知道用户已完成 MFA 验证，这可能会导致指标不是 100%。</span><span class="sxs-lookup"><span data-stu-id="e781d-198">If the claim is missing, Azure AD (and therefore Partner Center) will not know that the user has completed MFA verification and this can cause the metric to not be 100%.</span></span>

* <span data-ttu-id="e781d-199">**自定义控制** - 无法使用 Azure AD 自定义控制来确定用户是否已通过第三方 MFA 解决方案完成了 MFA 验证。</span><span class="sxs-lookup"><span data-stu-id="e781d-199">**Custom Control** - Azure AD Custom Control cannot be used to identify whether a user has completed MFA verification through a 3rd party MFA solution.</span></span> <span data-ttu-id="e781d-200">因此，对于 Azure AD（因而也包括合作伙伴中心）而言，通过自定义控制完成 MFA 验证的任何用户始终看上去并未完成 MFA 验证。</span><span class="sxs-lookup"><span data-stu-id="e781d-200">As a result, any user who has completed MFA verification through a custom control will always appear to Azure AD (and in turn Partner Center) as not having completed MFA verification.</span></span> <span data-ttu-id="e781d-201">如果可能，我们建议在与 Azure AD 集成时改用标识联合，而不要使用自定义控制。</span><span class="sxs-lookup"><span data-stu-id="e781d-201">Where possible, it is recommended that you switch to using Identity Federation as opposed to Custom Control when integrating with Azure AD.</span></span>
